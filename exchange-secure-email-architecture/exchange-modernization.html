Executive Summary

This whitepaper outlines the technical requirements and architectural impact of replacing legacy plaintext SMTP (Port 25) with secure alternatives in Exchange Server Subscription Edition (SE). While Exchange SE maintains support for legacy methods, 2025 security standards and Windows Extended Protection (EP) necessitate a shift toward Exchange Web Services (EWS) for mailbox management and SMTPS (Secure SMTP) for notifications.

⚠️ KEY FINDING: Disabling plaintext SMTP is a "breaking change." All applications must be updated to support TLS 1.2+ and Modern Authentication or be transitioned to a secure relay.

⚖️ Load Balancer Strategy Comparison

SMTPS (Port 587/465)

Strategy: L4 Passthrough + SNAT

Why?

  • Preserves Extended Protection Channel Binding
  • Direct client-to-Exchange TLS handshake
  • No certificate termination needed
  • Simplifies trust model via SNAT
Configuration:
• Virtual Service: L4
• SSL Profile: Passthrough
• SNAT: Enabled
• Health Monitor: TCP

EWS (HTTPS)

Strategy: L7 Bridging + Certificate Sync

Why?

  • Requires session persistence
  • Deep application health monitoring
  • HTTP header manipulation capabilities
  • Content-based routing if needed
Configuration:
• Virtual Service: L7
• SSL Profile: Bridging
• Certificate: Must match Exchange
• Health Monitor: /ews/healthcheck.htm
Feature SMTPS (L4 Passthrough) EWS (L7 Bridging)
SSL Handling Passthrough (no termination) Terminate and re-encrypt
Extended Protection ✅ Fully compatible ✅ Compatible (with cert sync)
Certificate Requirement Exchange only Avi + Exchange (must match)

🚀 Migration Phases (Zero-Downtime Strategy)

1

Coexistence

Duration: 30-60 days

Active Endpoints:

  • Port 25 (restricted IPs only)
  • Port 587 (new secure relay)
  • EWS/HTTPS (mailbox processing)
⚠️ Action: Deploy new connectors while maintaining legacy support
2

Migration

Duration: 60-90 days

Priority Order:

  1. High-risk/high-volume apps → EWS
  2. Simple relay apps → SMTPS
  3. Legacy/complex apps → Assess & update
📊 Goal: Port 25 traffic reduces from 100% → 0%
3

Decommission

Duration: 30 days validation

Validation Criteria:

  • Port 25 traffic = 0 for 30+ days
  • All apps migrated and validated
  • Security audit complete
  • Rollback plan documented
✅ Action: Disable Port 25 connector entirely

🎯 Security Impact Summary

Security Aspect Before (Legacy) After (2025)
Encryption ❌ Plaintext SMTP ✅ TLS 1.2+ enforced
Extended Protection ❌ Disabled ✅ Enforced on all endpoints
Trust Model ⚠️ 100+ individual server IPs ✅ Avi SE subnet (simplified)
Mailbox Operations ⚠️ IMAP copy-then-delete (2 steps) ✅ EWS native MoveItem (1 step)
Authentication ⚠️ Anonymous/basic auth ✅ Modern auth / certificate-based

✅ Final Recommendations

  1. Deploy EWS for all applications requiring "Inbox-to-Folder" movement. The native MoveItem operation is superior to IMAP's copy-then-delete pattern.
  2. Deploy SMTPS (L4 Passthrough) on Avi for all outbound relay needs. This preserves Extended Protection while simplifying configuration.
  3. Synchronize Certificates across Avi and Exchange to allow L7 Bridging for EWS without breaking Extended Protection.
  4. Consolidate Trust by using SNAT on the Load Balancer. This reduces Receive Connector management from 100+ IPs to a single subnet of Avi Service Engines.
  5. Follow the three-phase migration strategy to ensure zero downtime and proper validation at each stage.
  6. Maintain compliance with Exchange SE licensing requirements, including CALs for all authenticating entities and timely CU updates.
